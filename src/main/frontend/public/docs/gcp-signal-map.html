영어 신호어(signal word)를 보면
손이 먼저 서비스를 고르게

이 시험은 영어로 출제된다. 문제를 다 읽고 해석하는 게 아니라, 지문 속 트리거 영어 표현을 보는 순간 정답 서비스가 떠오르게 만드는 게 핵심이다. 아래 지도는 시험 전 범위를 신호어 → 개념 → 서비스 한 줄 흐름으로 그려 놓은 것이다. 함정(닮은꼴 서비스)까지 같이 본다.

1 · 지문에서 본다 "scale to zero" "pay per use"
2 · 개념으로 변환 serverless 완전관리 · 무과금
3 · 서비스가 떠오른다 Cloud Run
신호어 signal (영어 트리거) 서비스 service (정답) 함정 trap (닮은꼴 오답) 상단 검색창에 영어 표현을 넣으면 해당 카드만 남는다.
일치하는 신호어가 없습니다. 다른 영어 표현으로 검색해 보세요.

컴퓨트 선택

Compute Engine · GKE · Cloud Run · Cloud Functions

"어디에 실행할까"는 세 축으로 갈린다: 관리 부담(managed?), 0으로 축소되나(scale to zero?), 컨테이너/오케스트레이션이 필요한가. 아래 트랙에서 왼쪽 영어 표현이 보이면 오른쪽 서비스로 손이 가야 한다.

GKE 심화 — 모드 & 오토스케일링

Autopilot vs Standard · HPA · VPA · Cluster Autoscaler

GKE는 두 가지를 물어본다: 어떤 모드(mode)냐, 그리고 무엇을 스케일하냐. 오토스케일링 세 종류를 헷갈리면 반드시 틀린다 — 각각 스케일하는 대상이 다르다.

한눈에: 무엇을 스케일하나?
파란 사각형 = pod, 주황 박스 = node. HPA·VPA는 pod를, Cluster Autoscaler는 node를 건드린다.

GKE Autopilot

노드를 Google이 관리 · pod 단위 과금

네트워킹

Cloud NAT · IAP · Private Google Access · Interconnect · Shared VPC

여기서 제일 많이 실수한다. 핵심은 방향(direction)이다 — 들어오는(inbound) 연결인지, 나가는(outbound) 연결인지 먼저 판단하라. 그리고 "private VM(외부 IP 없음)"이라는 조건이 붙으면 선택지가 확 좁혀진다.

한눈에: 방향으로 갈린다 (inbound=IAP, outbound=NAT)
들어오는 SSH = IAP(초록, 왼→VM). 나가는 인터넷 = Cloud NAT(초록, VM→오른). NAT로 들어오려는 빨간 화살표는 차단.

IAP TCP forwarding

방향: inbound — 외부 → private VM
외부 IP 없이 SSH/RDP 접속. Google 프록시에서 IAM으로 신원 검증 → 최소 권한.
SSH to private VMno external IPleast privilege access

Cloud NAT

방향: outbound only — VM → 인터넷
외부 IP 없는 VM이 패치 다운로드 등 나갈 때. inbound는 절대 불가.
outbound internetdownload OS patchesno public IP
가르는 한 마디SSH로 들어가기(inbound) = IAP. 인터넷으로 나가기(outbound) = Cloud NAT. Cloud NAT는 inbound를 절대 허용하지 않는다 — 이걸 SSH에 쓰면 오답.

IAM & 리소스 계층

basic vs predefined vs custom · resource hierarchy · service accounts

"as restricted as possible / least privilege / most restrictive" 신호가 보이면 1단계로 basic role(Owner·Editor·Viewer)을 무조건 배제하고 시작한다. 그다음 요구된 동사(manage? deploy? view?)에 딱 맞는 predefined role을 고른다.

한눈에: 아래로 갈수록 권한이 좁다 = least privilege
"least privilege / as restricted as possible" 신호 → 맨 위 Basic부터 무조건 배제하고 시작.
"least privilege" 신호가 보이면 이 사다리로 내려간다
① Basic roles = roles/owner · editor · viewer — 범위가 프로젝트 전체라 거의 항상 오답. 먼저 버린다.
② Predefined roles → 서비스별 세분화 — 요구 동사에 맞춰 선택. 대부분 정답이 여기.
"manage" (IAM policy 포함) → ...admin · "deploy/develop" → ...developer · "read/view" → ...viewer
③ Custom roles → predefined도 넓을 때 필요한 권한만 조립하는 최종 수단.

Backup & Restore
가장 쌈. "cost is priority / data loss acceptable / RTO hours OK" 신호.
Cold standby
두 번째 리전 최소 인프라. 장애 시 scale-up 시간 필요.
Warm / Pilot light

lookalike traps · one-line tells

틀리는 문제의 절반은 "닮은꼴"에서 나온다. 각 쌍의 가르는 한 마디(the tell)만 손 노트에 옮겨 적어라.

닮은꼴 → 가르는 한 마디the tells전 범위
1Cloud NAT vs IAP — NAT=outbound만, IAP=inbound SSH. NAT는 inbound 절대 불가.
2Cluster Autoscaler vs Cloud Run — Cluster는 node 스케일, scale-to-zero 아님. zero면 Cloud Run.
3Bigtable vs Spanner — Bigtable=NoSQL, Spanner=relational+global+strong consistency.
4Dataproc vs Dataflow — Dataproc=기존 Spark/Hadoop 클러스터, Dataflow=서버리스 새 파이프라인.
5Cloud Armor vs Model Armor — Cloud=네트워크(DDoS/WAF), Model=LLM prompt/response.
6KMS vs Secret Manager — KMS=암호화 키, Secret Manager=비밀번호/API키.
7Deployment Manager vs Terraform — 둘 다 declarative지만 권장 표준=Terraform.
8Storage Transfer Service vs Transfer Appliance — 온라인(대역폭 충분) vs 오프라인(PB급·느린 망).
9Private Google Access vs Cloud NAT — Google API 전용 사설 vs 범용 인터넷 아웃바운드.
10HPA / VPA / Cluster Autoscaler — pod 개수 / pod 크기 / node 개수.
11basic role(Owner/Editor) vs predefined role — "least privilege"면 basic 먼저 배제.
12machine type 증설 vs RAG / Model Monitoring — 정확도 문제에 하드웨어 증설은 거의 항상 오답.
